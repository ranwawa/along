# PR Summary 规范要求

在创建 Pull Request 时，`prSummary` 参数必须包含完整的执行回顾，**禁止只写一两句简短描述**。

## 必须包含的章节

按照以下模板填充，每个章节都不可省略：

```markdown
fixes: #issue-id

## 修改内容
（逐文件列出具体做了什么改动）

## 执行步骤
（仅列出与业务逻辑相关的具体执行步骤）

## 影响范围
（说明改动对上下游模块的影响，是否有破坏性变更等）
```

## 示例

✅ **正确示例：**

```markdown
fixes: #28

## 修改内容
- `SharedBadge.tsx`：移除了"的分享"后缀，只展示分享者名称
- `SharedBadge.test.tsx`：更新测试中的预期文本

## 执行步骤
1. 定位到 SharedBadge 组件，分析问题根因
2. 修改组件代码，移除硬编码的后缀
3. 更新对应的测试用例
4. 运行 vitest 验证修改正确性

## 影响范围
- 仅影响首页列表中分享礼薄的展示
- 无破坏性变更
- 兼容现有数据
```

❌ **错误示例：**

- `修改了 SharedBadge 组件，去掉了"的分享"后缀` ← 缺少章节结构，过于简短
- 包含 `</think>`、`</function>` 等模型内部标记 ← 格式泄漏
