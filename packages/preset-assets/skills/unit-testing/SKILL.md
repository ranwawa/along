# 单元测试规范与指引

在本项目中，单元测试是确保代码质量和维护长期稳定性的核心手段。所有 Agent 必须严格遵循以下规范。

## 一、硬性要求 (Quality Gates)

1. **测试文件存在性**: 每个变更的 `.ts/.tsx` 源文件必须有对应的 `.test.ts(x)` 文件（同级）。不存在则必须**先创建测试文件**。
2. **80% 行覆盖率**: 运行测试时，针对本次变更的源文件必须达到 **80% 行覆盖率**。重点覆盖业务逻辑和边界情况，不要为凑覆盖率写无意义的测试。
3. **绿色通过**: 任何测试未通过时，严禁提交代码或发起 PR。严禁通过修改测试断言来“强行绿化”测试，除非原测试断言本身存在逻辑错误。

## 二、编写规范

- **命名与结构**: 测试描述必须使用**中文**，采用如下格式：

  ```typescript
  it("当 [输入/场景描述] 时，期望 [预期结果]", () => {
    // ...
  });
  ```

- **Mock 使用**: 尽量减少 `as any` 的使用。在构造 Mock 时，优先考虑类型安全的实现。
- **原子性**: 每个测试用例应保持独立，不依赖于其他测试的运行顺序。

## 三、执行提效策略

为节省 Token 并提高执行速度，建议遵循以下开发模式：

1. **局部运行**: 开发阶段仅运行与当前修改相关的测试文件：

   ```bash
   npx vitest run src/path/to/my-file.test.tsx
   ```

2. **减少输出噪声**: 使用 `--reporter=verbose` 并考虑重定向 stderr（如 `2>/dev/null`）以过滤无用的渲染日志，防止 context 被大量冗余日志占据。
3. **查阅 CI 日志**: 如果本地难以复现问题，指引模型使用 `gh run view` 查看失败的 GitHub Actions 详细日志。
4. **全量校验**: 在所有局部修复完成后，**必须运行一次全量测试**以确保没有导致逻辑回归。

## 四、自测 Checklist

- [ ] 本次修改的文件是否有对应测试文件？
- [ ] 变更行的覆盖率是否达到 80%？
- [ ] 是否所有测试均已通过？
