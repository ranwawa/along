我们要着手处理一个新的 Issue，其编号为：**$1**。

为了安全和规范地完成各项任务，你需要**严格按照以下工作流 (SOP)** 循序渐进地执行。

## 工作空间说明

- **你的工作目录**：当前 worktree 目录
- **Issue 数据文件**：`../issue.json`（相对于工作目录）
- **追踪文件位置**：`../todo.md`（由脚本自动维护，第三步除外）
- **状态文件位置**：`../status.json`（由脚本自动维护，你不需要手动更新）

**重要纪律**：

1. **强串行依赖**：前一步未成功前，严禁开始下一步
2. **真实响应为准**：收到工具调用**成功返回结果**后，才视为该步骤完成

## 自动化管理机制

- **`status.json`**：由各脚本（`branch-create`、`commit-push`、`pr-create`）在成功执行后**自动更新**
- **`todo.md`**：由各脚本在成功执行后**自动勾选**对应步骤，并附上时间戳和产出文件引用（第三步除外，需你手动更新）
- **产出文件**：每个脚本执行成功后会将详细产出保存到 `../step{N}-{脚本名}.md`

你无需手动更新 `status.json` 或 `todo.md`（第三步除外），专注于分析和写代码。

## 执行模式

请先读取模式文件确定本次执行范围：

```bash
cat ../.along-mode 2>/dev/null || echo "phase1"
```

- **`phase1`**（默认）：仅执行第一步和第二步。完成后正常退出，等待人工审批。
- **`phase2`**：跳过第一步和第二步（已完成），先读取 `../step2-issue-comment.md` 了解已审批的计划，然后从第三步开始执行。

## 强制工作流程

### 第一步：理解 Issue 并创建语义化分支

1. **读取 Issue 数据**：

   ```bash
   cat ../issue.json
   ```

   从 JSON 中提取标题、正文、标签等关键信息，理解需求本质（bug / feature / refactor / docs / ...）。

2. **生成语义化分支名**：

   参考 `branch-naming` SKILL，根据 Issue 类型和内容生成符合规范的分支名。格式：`<type>/issue-<N>-<short-description>`。

3. **创建分支**：

   ```bash
   along branch-create $1 <生成的分支名>
   ```

   该命令会自动完成：创建并切换分支 → 推送到远端 → 更新状态文件 → 勾选 todo。

### 第二步：分析代码库并制定实施计划

1. **深入分析代码**：

   - 根据 Issue 标题和正文，提取关键词，搜索相关代码
   - 使用 `Grep` 搜索关键词、函数名、错误信息
   - 使用 `Glob` 定位文件模式
   - 使用 `Read` 读取关键文件内容
   - 标记需要修改的文件和预计改动范围

2. **制定实施计划**：

   - 列出需要修改的文件及每个文件的预计改动
   - 明确依赖关系和修改顺序
   - 确定测试策略

3. **同步计划到 Issue 评论**：

   ```bash
   along issue-comment $1 "计划内容（Markdown 格式）" --step 2
   ```

   该命令会自动完成：发表评论 → 勾选 todo。

### 第三步：实施修复

**严格按计划执行**：

1. **分文件修改**：
   - 使用 `Edit` 工具修改代码
   - 每个文件修改后简要验证
   - 保持代码风格一致

2. **单元测试**：
   - 如涉及需要测试的代码变更，参考 `unit-testing` SKILL 编写或更新测试
   - 局部运行相关测试验证通过

3. **实时记录**：
   - 每完成一个修改单元，更新 `../todo.md` 第三步的进度

### 第四步：提交并推送代码

1. **编写提交信息**：

   参考 `conventional-commits` SKILL，编写符合规范的提交信息。格式：`<type>(<scope>): <中文描述>`。

2. **原子化提交并推送**：

   通过 `commit-push` 执行，它会自动处理 rebase 和推送。目标项目的 git hooks 会自动执行质量门禁（prettier、eslint、tsc、test）。

   - 单次提交：
     ```bash
     along commit-push --message "fix(scope): 描述" --files 文件1 文件2
     ```
   - 多次原子化提交：
     ```bash
     along commit-push --json '[{"message":"feat: first","files":["a.ts"]},{"message":"feat: second","files":["b.ts"]}]'
     ```

   该命令会自动完成：提交 → rebase → 推送 → 更新状态文件 → 勾选 todo。

3. **如果提交失败**：根据 hooks 返回的错误信息修复问题后重试，直到通过。

### 第五步：创建 PR 并更新状态

1. **撰写 PR 摘要**：

   参考 `pr-summary` SKILL，编写结构完整的 PR 描述。必须包含 `fixes: #$1`、修改内容、执行步骤、影响范围。

2. **创建 PR**：

   ```bash
   along pr-create $1 "PR 标题" "PR 详细摘要"
   ```

   该命令会自动完成：创建 PR → 更新状态文件 → 勾选 todo。

   PR 创建后状态保持 `running`，等待审核与合并。只有 PR 被合并到默认分支后，状态才会变为 `completed`。

## 终止与清理

### 正常完成

任务完成后，状态文件会保留（归档到 `archive/` 目录），便于历史追溯。

### 异常终止

如果任务需要提前终止（无法完成或用户取消）：

```bash
along issue-status $1 error --message="终止原因"
```

### 强制重新开始

```bash
# 清理旧的工作空间
along cleanup $1 --force
# 重新初始化（由 along run 自动完成）
```

## 模型职责边界

### 你必须做的（非确定性任务）

1. **理解与分析**：理解 Issue 需求，分析代码，制定策略
2. **决策**：判断修改方案，选择实现方式，生成语义化分支名
3. **代码修改**：编写和修改代码
4. **验证**：检查修改是否正确解决问题
5. **沟通**：通过评论等方式与团队沟通

### 你不应该做的（由工具/脚本处理）

1. **工作空间初始化**：不手动创建 worktree，这是 `along run` 的职责
2. **状态文件维护**：不直接读写 `status.json`，由脚本自动更新
3. **进度追踪**：不手动更新 `todo.md`（第三步除外），由脚本自动勾选
4. **Git 底层操作**：不直接执行 `git worktree add/remove`
5. **目录管理**：不创建/删除 worktree 目录结构
6. **质量门禁**：不手动执行 prettier/eslint/tsc，由 git hooks 自动处理

遵循以上边界，专注于你擅长的分析与实现工作，将确定性操作交给工具处理。
