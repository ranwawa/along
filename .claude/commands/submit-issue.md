你需要帮助用户在当前仓库中提交一个 GitHub Issue。

## 核心原则

- **只描述问题，不分析原因，不提供解决方案**
- 聚焦于：背景 → 问题描述（含证据） → 期望行为
- 用中文撰写，技术术语保持英文

## 信息收集

根据上面用户的描述提取以下信息。如果已经提供了足够的信息，直接组织并提交；如果关键信息缺失，简短追问。

1. **问题类型**：bug / feature / improvement / docs
2. **背景环境**：涉及的功能模块、使用场景、相关版本、运行环境等上下文信息
3. **问题描述**：具体发生了什么 / 缺少什么能力，以及支撑问题的证据（错误日志、截图、跟踪记录、相关链接、最小可复现示例等）
4. **期望行为**：正确的/理想的表现应该是什么

## Issue 模板

根据问题类型选择对应的标题前缀和标签：

| 类型 | 标题格式 | 标签 |
|------|---------|------|
| bug | `[Bug] 简短描述` | `bug` |
| feature | `[Feature] 简短描述` | `enhancement` |
| improvement | `[Improvement] 简短描述` | `enhancement` |
| docs | `[Docs] 简短描述` | `documentation` |

## 提交格式

使用以下 Markdown 结构组织 Issue body。**背景模块不描述问题本身**，只提供环境上下文；**问题描述模块**承载所有问题现象和证据。

**Bug 类型：**

```markdown
## 背景

- **功能模块**：[涉及的模块/命令/组件]
- **使用场景**：[在什么业务流程或操作中触发]
- **运行环境**：[OS、Runtime 版本、相关工具版本等，仅列与问题相关的]
- **相关版本**：[软件版本、依赖版本等]

## 问题描述

[清晰描述当前实际发生的情况]

**证据：**

[以下根据实际情况选择性提供，尽量多提供以便快速定位根因]

- 错误日志 / 终端输出（用代码块包裹）
- 截图
- 跟踪记录 / 调试信息
- 相关链接（PR、Commit、文档等）
- 最小可复现步骤：
  1. [步骤1]
  2. [步骤2]
  3. ...

## 期望行为

[描述正确的/理想的表现应该是什么]
```

**Feature / Improvement 类型：**

```markdown
## 背景

- **功能模块**：[涉及的模块/命令/组件]
- **使用场景**：[在什么业务流程或操作中需要此功能/改进]
- **当前版本**：[相关版本信息，可选]

## 问题描述

[描述当前缺少什么能力，或现有行为的不足之处]

**证据：**

[可选，根据实际情况提供]

- 相关链接（Issue、PR、文档等）
- 截图 / 示意图
- 当前行为的日志或输出

## 期望行为

[描述希望实现的功能或改进后的表现]
```

## 执行步骤

1. 根据用户描述组织 Issue 内容
2. 确定合适的标题、标签
3. 使用 `gh issue create` 提交：

```bash
gh issue create --title "<标题>" --label "<标签>" --body "<Issue body>"
```

4. 返回创建的 Issue 链接给用户

## 注意事项

- 标题要简洁有力，一眼看出问题所在
- body 中不要包含「可能是因为...」「建议修改...」等分析性内容
- 如果用户提供了代码片段或日志，原样保留在「问题描述」的证据部分
- 背景模块只放环境上下文，不描述问题现象
- 尽量在问题描述中附带证据，帮助处理人快速定位根因
