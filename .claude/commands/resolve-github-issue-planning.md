我们要着手处理一个新的 Issue，其编号为：**$1**。

为了安全和规范地完成各项任务，你需要**严格按照以下工作流 (SOP)** 循序渐进地执行。

## 工作空间说明

- **你的工作目录**：当前 worktree 目录
- **Issue 数据文件**：`../issue.json`（相对于工作目录）
- **追踪文件位置**：`../todo.md`（由脚本自动维护）
- **状态管理**：由脚本自动维护（存储在 SQLite 数据库中，你不需要手动更新）

**重要纪律**：

1. **强串行依赖**：前一步未成功前，严禁开始下一步
2. **真实响应为准**：收到工具调用**成功返回结果**后，才视为该步骤完成

## 自动化管理机制

- **会话状态**：由各脚本（`branch-create`、`commit-push`、`pr-create`）在成功执行后**自动更新**到数据库
- **`todo.md`**：由各脚本在成功执行后**自动勾选**对应步骤，并附上时间戳和产出文件引用
- **产出文件**：每个脚本执行成功后会将详细产出保存到 `../step{N}-{脚本名}.md`

你无需手动更新会话状态或 `todo.md`，专注于分析和写代码。

## 强制工作流程

### 第一步：理解 Issue 并创建语义化分支

1. **读取 Issue 数据**：

   ```bash
   cat ../issue.json
   ```

   从 JSON 中提取标题、正文、标签等关键信息，理解需求本质（bug / feature / refactor / docs / ...）。

2. **生成语义化分支名**：

   参考 `branch-naming` SKILL，根据 Issue 类型和内容生成符合规范的分支名。格式：`<type>/issue-<N>-<short-description>`。

3. **创建分支**：

   ```bash
   along branch-create $1 <生成的分支名>
   ```

   该命令会自动完成：创建并切换分支 → 推送到远端 → 更新状态文件 → 勾选 todo。

### 第二步：分析代码库并制定实施计划

1. **深入分析代码**：

   - 根据 Issue 标题和正文，提取关键词，搜索相关代码
   - 使用 `Grep` 搜索关键词、函数名、错误信息
   - 使用 `Glob` 定位文件模式
   - 使用 `Read` 读取关键文件内容
   - 标记需要修改的文件和预计改动范围

2. **制定实施计划**：

   - 列出需要修改的文件及每个文件的预计改动
   - 明确依赖关系和修改顺序
   - 确定测试策略

3. **同步计划到 Issue 评论**：

   ```bash
   along issue-comment $1 "计划内容（Markdown 格式）" --step 2
   ```

   该命令会自动完成：发表评论 → 勾选 todo。

完成第一步和第二步后，正常退出，等待人工审批。审批通过后由脚本自动切换到 Implementation 阶段。

## 模型职责边界

### 你必须做的（非确定性任务）

1. **理解与分析**：理解 Issue 需求，分析代码，制定策略
2. **决策**：判断修改方案，选择实现方式，生成语义化分支名
3. **沟通**：通过评论等方式与团队沟通

### 你不应该做的（由工具/脚本处理）

1. **工作空间初始化**：不手动创建 worktree，这是 `along run` 的职责
2. **状态维护**：不直接操作会话状态数据库，由脚本自动更新
3. **进度追踪**：不手动更新 `todo.md`，由脚本自动勾选
4. **Git 底层操作**：不直接执行 `git worktree add/remove`
5. **目录管理**：不创建/删除 worktree 目录结构
6. **实施阶段**：不进入实施阶段，此阶段仅做规划和分析

遵循以上边界，专注于你擅长的分析与规划工作，将确定性操作交给工具处理。
