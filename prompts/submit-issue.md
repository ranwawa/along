---
description: 快速提交高质量 GitHub Issue，聚焦问题描述而非解决方案
---

你需要帮助用户在当前仓库中提交一个 GitHub Issue。

## 核心原则

- **只描述问题，不分析原因，不提供解决方案**
- 聚焦于：背景 → 现象 → 期望行为
- 用中文撰写，技术术语保持英文

## 信息收集

在提交 Issue 之前，你需要从用户的描述中提取以下信息。如果用户已经提供了足够的信息，直接组织并提交；如果关键信息缺失，简短追问。

1. **问题类型**：bug / feature / improvement / docs
2. **背景**：在什么场景下遇到/需要这个问题
3. **现象**（bug 必填）：具体发生了什么，包含错误信息、截图等
4. **期望行为**：正确的/理想的表现应该是什么
5. **复现步骤**（bug 建议提供）：最小化的操作步骤

## Issue 模板

根据问题类型选择对应的标题前缀和标签：

| 类型 | 标题格式 | 标签 |
|------|---------|------|
| bug | `[Bug] 简短描述` | `bug` |
| feature | `[Feature] 简短描述` | `enhancement` |
| improvement | `[Improvement] 简短描述` | `enhancement` |
| docs | `[Docs] 简短描述` | `documentation` |

## 提交格式

使用以下 Markdown 结构组织 Issue body：

```markdown
## 背景

[描述遇到问题的场景和上下文]

## 当前行为

[描述当前实际发生的情况，包含具体的错误信息、日志或截图]

## 期望行为

[描述正确的/理想的表现应该是什么]

## 复现步骤

1. [步骤1]
2. [步骤2]
3. ...

## 环境信息

- OS: [操作系统]
- 版本: [相关版本信息]
```

对于 feature/improvement 类型，可以省略「当前行为」和「复现步骤」，改用：

```markdown
## 背景

[描述需求的场景和动机]

## 期望行为

[描述希望实现的功能或改进]

## 补充说明

[可选：使用示例、参考链接等]
```

## 执行步骤

1. 根据用户描述组织 Issue 内容（标题、标签、body）
2. **确认方案**：将组织好的 Issue 内容展示给用户，征求确认
3. 用户确认后，使用 `gh issue create` 提交：

```bash
gh issue create --title "<标题>" --label "<标签>" --body "<Issue 内容>"
```

4. 返回创建的 Issue 链接给用户

## 确认话术

展示 Issue 草稿时，使用以下话术征求确认：

```
已组织好 Issue 草稿，请确认后提交：

---

**标题**：[标题内容]
**标签**：bug / enhancement / documentation

**正文**：
[正文内容]

---

确认提交？回复「是」或「确认」即可提交，回复「修改」并说明需要修改的部分。
```

## 注意事项

- 标题要简洁有力，一眼看出问题所在
- body 中不要包含「可能是因为...」「建议修改...」等分析性内容
- 如果用户提供了代码片段或日志，原样保留在「当前行为」中
- 环境信息只在 bug 类型中需要，且只包含与问题相关的信息
- **必须等待用户确认后才能提交**，不要在用户未确认的情况下直接提交
