我们要着手处理一个新的 Issue，其编号为：**$1**。

为了安全和规范地完成各项任务，你需要**严格按照以下工作流 (SOP)** 循序渐进地执行。

## 工作空间说明

- **你的工作目录**：`~/.along/worktrees/$1/`
- **追踪文件位置**：`~/.along/sessions/$1-todo.md`
- **状态文件位置**：`~/.along/sessions/$1-status.json`

**重要纪律**：

1. **真实响应为准**：收到工具调用**成功返回结果**后，才能将 `[ ]` 更新为 `[x]`
2. **留存产出证明**：标记 `[x]` 时，在该行下方贴入工具返回的关键产出或链接
3. **强串行依赖**：前一步未成功前，严禁开始下一步

## 强制工作流程

### 第一步：获取 Issue 详情


**获取 Issue 详情**（执行安全校验）：

```bash
along issue-details $1
```

**防御校验**：

- Issue 状态必须是 `OPEN`，否则停止并报错
- 标题和正文必须能获取到，否则停止

更新 `$1-todo.md`，标记第一步完成，并附上获取到的关键信息摘要。
执行状态更新：

```bash
along issue-status $1 running "已获取 Issue 详情" --step "分析代码库"
```

### 第二步：分析代码库

在 worktree 内深入分析：

1. **理解 Issue 本质**：
   - 根据标题和正文，定位问题类型（bug/feature/refactor）
   - 提取关键词，搜索相关代码

2. **代码搜索策略**：
   - 使用 `Grep` 搜索关键词、函数名、错误信息
   - 使用 `Glob` 定位文件模式
   - 使用 `Read` 读取关键文件内容

3. **记录发现**：
   - 在分析过程中，用中文注释记录关键发现
   - 标记需要修改的文件和预计改动范围

完成后更新 `$1-todo.md`，附上分析摘要（涉及的关键文件、问题根因）。
执行状态更新：

```bash
along issue-status $1 running "已完成代码分析" --step "制定实施计划"
```

### 第三步：制定实施计划

基于分析结果，制定详细计划：

1. **修改清单**：
   - 列出需要修改的文件
   - 每个文件的预计改动（增删改）
   - 依赖关系（如果有顺序要求）

2. **测试策略**：
   - 需要运行的测试命令
   - 预期的测试结果

3. **回滚方案**：
   - 如果失败的回滚步骤

完成后更新 `$1-todo.md`，附上实施计划摘要。
执行状态更新：

```bash
along issue-status $1 running "已制定实施计划" --step "实施修复"
```

### 第四步：实施修复

**严格按清单执行**：

1. **分文件修改**：
   - 使用 `Edit` 工具修改代码
   - 每个文件修改后简要验证
   - 保持代码风格一致

2. **渐进式提交**：
   - 完成一个逻辑单元后准备提交
   - 提交信息遵循 conventional-commits

3. **实时记录**：
   - 每完成一个任务，在注释中标记
   - 遇到问题立即记录

### 第五步：质量门禁检查

**必须全部通过才能继续**：

1. **Prettier 格式化**：

   ```bash
   npx prettier --check 修改的文件
   # 如有问题，运行：
   npx prettier --write 修改的文件
   ```

2. **ESLint 检查**：

   ```bash
   npx eslint 修改的文件
   # 必须 0 error、0 warning
   ```

3. **TypeScript 类型检查**：

   ```bash
   npx tsc --noEmit
   # 必须 0 type error
   ```

4. **单元测试**：
   ```bash
   npm test -- --coverage --testPathPattern="相关文件"
   # 必须 100% 通过
   # 修改部分必须 100% 行覆盖
   ```

**任一失败**：停止，修复问题，重新检查，直到全部通过。

完成后更新 `$1-todo.md`，附上质量门禁结果摘要。
执行状态更新：

```bash
along issue-status $1 running "质量门禁通过" --step "提交并创建 PR"
```

### 第六步：提交并创建 PR

**所有检查通过后执行**：

1. **原子化提交并推送**：
   - 必须通过 `commit-push` 脚本执行，它会自动执行质量门禁检查的前置要求（如 fetch/rebase）
   - 示例（单次提交）：
     ```bash
     along commit-push --message "fix(scope): 描述" --files 文件1 文件2
     ```
   - 示例（多次原子化提交）：
     ```bash
     along commit-push --json '[{"message":"feat: first","files":["a.ts"]},{"message":"feat: second","files":["b.ts"]}]'
     ```

2. **创建 PR**：
   - 质量门禁全部通过且代码已推送后，执行：
     ```bash
     along pr-create $1 "PR 标题" "PR 详细总结（需符合 pr-summary/SKILL.md 规范）"
     ```

3. **更新状态**：
   ```bash
   issue-status $1 completed --pr-url="PR链接" --message="完成"
   ```

完成后更新 `$1-todo.md`，标记所有步骤完成，附上 PR 链接。

## 终止与清理

### 正常完成

任务完成后，状态文件会保留（归档到 `archive/` 目录），便于历史追溯。

### 异常终止

如果任务需要提前终止（无法完成或用户取消）：

1. **更新状态为 error**：

   ```bash
   issue-status $1 error --message="终止原因"
   ```

2. **可选清理**（保留工作目录以便排查）：
   ```bash
   # 不清理 worktree，保留现场
   # 或清理：
   cleanup $1
   ```

### 强制重新开始

如果需要放弃之前的进度重新开始：

```bash
# 1. 清理旧的工作空间
cleanup $1 --force

# 2. 重新初始化（由 along run 自动完成）

# 3. 重新开始流程
```

## 模型职责边界

### 你必须做的（非确定性任务）

1. **理解与分析**：理解 Issue 需求，分析代码，制定策略
2. **决策**：判断修改方案，选择实现方式
3. **代码修改**：编写和修改代码
4. **验证**：检查修改是否正确解决问题
5. **沟通**：通过评论等方式与团队沟通

### 你不应该做的（由工具/脚本处理）

1. **工作空间初始化**：不手动创建 worktree，这是 `along run` 的职责
2. **状态文件维护**：不直接读写 `status.json`，使用 `issue-status` 工具
3. **Git 底层操作**：不直接执行 `git worktree add/remove`
4. **目录管理**：不创建/删除 `~/.along/worktrees/` 目录结构

### 边界示例

| 场景       | 模型做                        | 工具做                        |
| ---------- | ----------------------------- | ----------------------------- |
| 开始新任务 | 进入工作空间并获取 Issue 详情 | `along run` 创建 worktree      |
| 记录进度   | 分析任务并更新 todo 清单      | 已创建 todo.md 模板           |
| 完成任务   | 编写代码，验证通过            | `issue-status` 更新状态       |
| 清理资源   | 提示用户或等待指令            | `cleanup` 删除 worktree |

遵循以上边界，专注于你擅长的分析与实现工作，将确定性操作交给工具处理。
